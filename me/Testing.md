### Situation

團隊原本只有針對共用 library 寫 unit test，產品主要功能幾乎沒有測試覆蓋。隨著功能越來越多，問題開始浮現：

- **改 A 壞 B**：修改某個 API 的邏輯後，意外影響到其他相依的 API，直到上線才被發現
- **Edge case 漏網**：缺乏系統性測試，許多邊界條件（例如金額為零、空值、併發情境）只在 production 才暴露出來
- 團隊對部署越來越沒信心，每次上線都像在拆炸彈

### Task
由我主導建立產品層級的測試體系，目標是讓主要業務流程都有測試保護，並在 CI pipeline 中設置品質關卡，確保有問題的程式碼不會進入 production。

### Action

**1. 分層測試策略設計**

- **Unit Test**：針對核心商業邏輯（金額計算、手續費、分潤邏輯）撰寫，確保單一函式的正確性，特別關注邊界值和異常輸入
- **Integration Test**：針對 API endpoint 撰寫，模擬完整的 request → 業務處理 → DB 操作 → response 流程，使用測試資料庫驗證資料正確性
- **E2E Test**：針對關鍵的跨服務業務流程（例如使用者下注 → 結算 → 派獎的完整鏈路）撰寫端對端測試

**2. 測試框架與工具選擇**

- 統一使用 **Jest** 作為測試框架，搭配 supertest 做 HTTP API 測試
- 建立共用的 test utilities：DB seeder（快速建立測試資料）、mock factory（統一管理外部依賴的 mock）

**3. CI Quality Gates 設置**

- 在 Jenkins pipeline 中加入測試階段，設定以下硬性規則：
    - **Coverage 門檻 80%**：低於 80% 的 PR 無法合併
    - **全部測試必須通過**：任何一個 test case 失敗，pipeline 直接中斷
- PR merge 流程：程式碼提交 → CI 自動跑測試 + 覆蓋率檢查 → 通過後才能 merge 進主分支

**4. 測試文化推動**

- 在團隊內建立「新功能必須附帶測試」的規範
- 將主要業務流程（金融報表、交易處理、客服通訊）優先補上測試

### Result

- 測試覆蓋率從接近 **0%（產品層級）提升到 80% 以上**
- 系統穩定性顯著提升，**上線後的 bug 回報大幅減少**，團隊對部署的信心明顯增加
- CI quality gate 有效攔截了多次潛在的 regression — 改 A 壞 B 的情況幾乎不再發生
- 測試成為團隊日常開發流程的一部分，新功能交付時已包含測試

- **題號**: 110
- **核心標籤**: `樹 (Tree)`, `遞迴 (Recursion)`, `深度優先搜尋 (DFS)`
- **題目精髓**: 這是一道經典的遞迴設計題。它教導我們如何透過設計一個**身兼二職 (dual-purpose)** 的輔助函式，在一次遍歷中同時完成「檢查條件」和「計算數值」兩項任務，從而避免低效率的重複計算。
- **關鍵思想：先定義遞迴「契約」(Define the Contract First)**
    - 直接遞迴 `isBalanced` 會導致在每個節點重複計算高度，效率低下。
    - **最佳解法**是設計一個輔助函式 (e.g., `checkHeight`)，並為其制定一個清晰的「契約」：
> 如果子樹是平衡的，就回傳它的高度 (一個 >= 0 的整數)；如果子樹不平衡，就回傳一個特殊信號 -1。
    - 這個契約是整個演算法的設計藍圖和指導原則。
- **演算法模式 (身兼二職的輔助函式)**
    - 這個輔助函式採用**後序遍歷 (Post-order Traversal)** 的順序，即 `左 -> 右 -> 中`。
    - **1. 遞迴呼叫 (處理子節點)**:
        - 先呼叫 `left_info = checkHeight(node.left)` 獲取左子樹的資訊。
        - 再呼叫 `right_info = checkHeight(node.right)` 獲取右子樹的資訊。
    - **2. 檢查失敗信號 (處理當前節點 - Part 1)**:
        - 檢查子節點的回報：只要 `left_info` 或 `right_info` 中有一個是 `1`，就代表子樹已經不平衡。此時應「提前退出」，立即向上回傳 `1`。
        - 檢查當前節點的條件：`if abs(left_info - right_info) > 1`，如果不滿足平衡條件，則從當前節點開始，回傳 `1`。
    - **3. 回報成功結果 (處理當前節點 - Part 2)**:
        - 如果所有檢查都通過，代表到目前為止一切平衡。
        - 履行契約的另一半：回傳當前子樹的高度 `1 + max(left_info, right_info)`。
    - 主函式 `isBalanced` 只需呼叫這個輔助函式，並判斷最終結果是否為 `1` 即可。
- **複雜度分析**:
    - **時間複雜度: O(N)**
        - N 是樹的節點總數。因為每個節點都只被訪問一次，所以是線性時間。
    - **空間複雜度: O(H)**
        - H 是樹的高度。空間來自遞迴呼叫堆疊的最大深度。
        - **平均情況 (平衡樹)**：H ≈ log(N)，空間為 O(log N)。
        - **最壞情況 ( skewed tree)**：H = N，空間為 O(N)。
- **takeaway**:
> 當遞迴函式需要回報的資訊（例如：高度和是否平衡），比題目規定的主函式簽名（例如：只能回傳布林值）更豐富時，就要果斷使用「輔助函式」。透過制定一個巧妙的「契約」，利用特殊回傳值（如 -1）來傳遞失敗信號，是解決此類問題最高效、最優雅的模式。

## 💡 面試實戰金句 (Elevator Pitch)

> 「在設計千萬級用戶的 API Gateway 時，單純的 Session 會導致 Redis OOM 與嚴重的 I/O 瓶頸；而純 JWT 雖然擴展性極佳，卻會引發『無法強制撤銷』的資安漏洞。
> 因此，我會採用 **Hybrid JWT (Stateful JWT)** 架構。利用 JWT 處理 99% 的熱路徑驗證與微服務間的身分傳遞 (Trust Boundary)；同時利用 Redis 維護一個極輕量的 Token 黑名單 (Blocklist) 來處理強制登出。這種『冷熱路徑分離』的設計，能在效能擴容與資安控管之間取得最完美的平衡。」

## 1. 傳統派：Cookie & Session (有狀態架構 Stateful)

最直覺、最傳統的做法。伺服器負責記住所有人，客戶端只拿一張「號碼牌」。

- **運作機制**：
    1. 登入成功後，伺服器在記憶體 (或 Redis) 中存入該使用者的資料（白名單/活躍清單）。
    2. 伺服器發放一個無意義的亂碼 `Session_ID`。
    3. 客戶端將 `Session_ID` 存在 **Cookie** 裡，每次發送 Request 都帶著它。
    4. 伺服器收到後，去 Redis 查詢這個 ID 對應誰。
- **✅ 優點**：
    - **絕對的控制權**：想踢掉誰（強制登出），只要從 Redis 把該 Session 刪除，下一秒他就會被擋在門外。
- **❌ 致命缺點 (擴展性災難)**：
    - **記憶體怪獸**：如果有一億個活躍用戶，Redis 要存一億筆資料，極度昂貴。
    - **I/O 瓶頸**：每次 API 請求都要去 Redis 查資料，I/O 延遲高。
    - **微服務不友善**：API Gateway 查完 Redis 後，後面的 Timeline Service、User Service 如果需要權限資料，難道又要去戳一次 Redis？（會引發 Redis 癱瘓）。

## 2. 理想派：純 JWT (無狀態架構 Stateless)

為了解決 Session 的擴展性問題而生，試圖打造完美的「無狀態」烏托邦。

- **運作機制**：
    1. 登入成功後，伺服器把使用者資訊 (如 `User_ID`, `Role`) 寫成一份 JSON。
    2. 伺服器用私鑰 (Secret/Private Key) 對這份 JSON 進行**密碼學簽章 (Signature)**，生成 JWT。
    3. 伺服器不存任何東西，直接把 JWT 交給客戶端。
    4. 客戶端每次請求帶著 JWT，API Gateway 只用 CPU 計算簽章是否正確，正確就放行。
- **✅ 優點**：
    - **零記憶體負擔**：伺服器不存狀態，Redis 負載趨近於零。
    - **擴容極簡**：Gateway 只要加機器，CPU 算力就能無限水平擴展。
    - **微服務完美接力**：Gateway 驗證完 JWT 後，直接把 Payload 傳給後端微服務，後端無條件信任，完全不需要連資料庫。
- **❌ 致命缺點 (資安核彈)**：
    - **無法提前撤銷 (No Revocation)**：如果使用者的筆電被偷，他在手機上按下「強制登出該筆電」。系統根本無能為力，因為 JWT 不在伺服器手上，只要 JWT 還沒過期，小偷就能繼續發送請求！

## 3. 實戰派：Hybrid JWT (混合架構 - 大廠終極解法)

結合了 JWT 的高效能與 Session 的安全性。這正是 Twitter (X)、Netflix 等巨頭的標準作法。

- **核心哲學：冷熱路徑分離 (Hot Path vs. Cold Path)**
    - **🔥 熱路徑 (每秒百萬次 API 請求)**：極限壓縮成本。API Gateway **只用 CPU 驗證 JWT 簽章**，並查詢極小的「Redis 黑名單」。
    - **🧊 冷路徑 (久久查一次的裝置清單)**：把完整的「活躍裝置清單 (Active Sessions)」存在傳統資料庫 (如 PostgreSQL) 中，只有使用者點進設定頁面時才去撈取。
- **運作機制與強制登出流程**：
    1. **日常請求 (極速通關)**：使用者帶著 JWT 來到大門口。API Gateway 驗證簽章 ➔ 查詢 Redis 黑名單 (Blocklist) ➔ 沒在名單內 ➔ 放行。
    2. **觸發登出 (冷熱交會)**：使用者按下「登出被偷的筆電」。
    3. **寫入黑名單**：系統將那台筆電的 JWT 專屬編號 (`jti`)，立刻寫入 **Redis 黑名單**。
    4. **成功攔截**：小偷用那台筆電發出請求 ➔ API Gateway 驗證簽章正確 ➔ 查詢 Redis 黑名單 ➔ **發現被 ban 了** ➔ 回傳 `401 Unauthorized` 踢出！
- **✅ 優點**：
    - 完美保留了 JWT 微服務傳遞與 CPU 驗證的高效能。
    - Redis 記憶體使用量暴降：從「存一億人的白名單」變成「只存幾萬個被盜/強制登出的黑名單」。

### Q1: 你如何決定 Kubernetes Pod 的 Resource Request 和 Limit 該設多少？

> 我不是憑感覺設的，我是看數據決定的。
我在 Kibana 建立了監控儀表板，觀察服務在P50及P95下的 Memory Usage。
例如，我觀察到 bms-management P50佔用約 165 MiB、P95 占用約 250。
> - **Request 設定：** 我會設在 **180~200 MiB**（略高於平均值），確保 Pod 能被調度到資源足夠的節點。
> - **Limit 設定：** 我會設在 **300~350 MiB**（約平均值的 1.5~2 倍），容許流量突波，但防止它吃光節點記憶體導致 OOM。

### Q2: 你怎麼判斷一個服務有沒有 Memory Leak (記憶體洩漏)？

> 我會觀察 Kibana 上的 Memory Usage 趨勢線。
> - **健康狀態：** 如同我目前的儀表板，曲線是**平穩的水平線**，或者呈現**鋸齒狀**（Sawtooth pattern，代表 GC 垃圾回收機制正常運作，記憶體有升有降）。
> - **Memory Leak：** 曲線呈現**階梯式上升 (Staircase)** 或 **45度角持續上升**，且長時間沒有下降回原本水平。一旦發現這種趨勢，我就會使用 Profiling 工具介入排查。

### Q3: 你的 CPU 圖表顯示只有 5m，這代表什麼？是不是資源浪費？

> 是的，5m (0.005 Core) 代表該服務目前非常閒置。
這給了我兩個優化方向：
> 1. **資源重分配 (Right-sizing)：** 如果這是常態，我可以降低該 Pod 的 CPU Request (例如設為 50m 或 100m)，將節省下來的 CPU 資源分給其他更吃重的運算任務。
> 2. **合併服務：** 如果多個微服務都這麼閒，考量維運成本，也許可以評估是否過度拆分服務。

### Q4: 為什麼你要在 Kibana 寫 Script 轉單位？不能直接看 Bytes 嗎？

> 建議回答 (Communication):
這是為了團隊溝通與準確性。
「173,015,040 Bytes」對人類來說很難一眼看出是多大，但「165 MiB」很直觀。
此外，K8s 的 YAML 設定檔通常使用 Mi 或 m 為單位。我在儀表板上統一單位，能減少我們在設定 metrics-server 或 HPA 時的換算錯誤風險。

### **遇到最難/有成就的技術性問題，以及你如何利用邏輯解決它**

situation: 我們因應客戶需求在客服系統上增加了 AI 功能，想藉由 AI 節省人力成本，達成類似 chatGPT 那樣聊天機器人一問一答的效果。首先 RAG 是一個在大型語言模型輸出的優化，它叫做檢索增強生成，相比於重新訓練 model 本身，他可以直接參考一個外部最新的資訊，來讓 model 向使用者提供更準確地答案，來改善這種生成式 AI 的品質。

task: 那其中遇到的困難，在我們使用的 openAI  gpt3.5 這個 model 下，在 RAG 上的效果非常的糟，可以說只有10~20%的準確度，因此要在不升級 model 的情況下改善這部分，因為升級後的價格相差了10幾倍。

action:  我找了很多資料，也測試了很久，後來想到一個方法是自己做一個 RAG ，並把他跟 gpt3.5 結合，也就是說把流程給拆分，讓可控制的部分多一些，那剛好網路上有技術文章或影片在分享，因此跟PM及主管討論後就決定使用這個方法。

result: 經過了一連串的奮鬥，最後終於成功讓準確度提升至了至少80%，主管及客戶也都很滿意這項優化，這是最有成就的部分。

### 如何協作/開發流程

我們團隊目前組成為1６個人，5名前端、5名後端、3名PM、3名PM

團隊上目前是使用 scrum 敏捷開發，每天會有10~15分鐘的stand up meeting，同步大家的工作進度，然後每周會有一次大型的會議，團隊會展示這一周完成的功能，以及討論下周要開發的內容，或是有需要改進的地方也會在這時提出來

工程上是使用 git flow 來協作開發 ，在有新功能確認前 PM 會先跟我及前端討論，那我會判斷這個功能的可行性，或是流程有哪裡需要修改的，之後會去思考如果新增這功能會不會影響到系統其他地方，再把這些工作任務切分成一塊一塊的，進行時程規劃、分派，如果我在開發的過程中有想到更好的做法，或是邏輯上有奇怪的地方，都會去詢問 PM 及 manager ，以免到時做出來的跟實際想要的不一樣，造成時程上的 delay。一連串的開發、測試都ok後，就會使用 CI/CD 工具部屬，讓 PM 、客戶做測試，以上是開發流程。

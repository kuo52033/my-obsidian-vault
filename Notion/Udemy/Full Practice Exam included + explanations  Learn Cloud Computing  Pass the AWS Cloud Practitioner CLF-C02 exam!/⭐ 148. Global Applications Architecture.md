## Overview

When designing a global application on AWS, you must consider:

- **High availability (HA)**
- **Global latency**
- **Complexity of deployment**
- **Replication model (single region vs multi-region)**

AWS offers several architectures depending on availability and latency requirements.

---

## 1. Single Region, Single Availability Zone

**Characteristics**

- Only 1 AZ and 1 Region
- No high availability
- Very poor global latency (users far away = slow)
- Lowest complexity

**Use cases**

- Development
- Very early prototype
- Non-critical systems

---

## 2. Single Region, Multi-AZ

**Characteristics**

- Uses 2+ AZs within the same region
- **High availability** (AZ failover)
- **Still no global latency improvement**
- Moderate complexity

**Use cases**

- Standard production workloads
- Most web applications requiring fault tolerance

---

## 3. Multi-Region — Active/Passive

**Characteristics**

- One region = **active (read + write)**
- Other region(s) = **passive (read-only or standby)**
- Data replication from active → passive
- **Better global read latency**
- **Writes still go to one region → global write latency is high**
- Higher complexity

**Use cases**

- Disaster recovery (DR)
- Global read access, but centralized writes
- Example AWS service: **RDS Multi-Region Read Replicas**

---

## 4. Multi-Region — Active/Active

**Characteristics**

- All regions can **read and write**
- Full bi-directional replication
- **Best global read + write latency**
- Highest complexity (conflict resolution, data consistency)

**Use cases**

- Real global applications
- Mission-critical systems requiring milliseconds latency worldwide
- Example AWS service: **DynamoDB Global Tables**

---

## Summary Table

| Architecture Type | HA | Global Latency | Write Latency | Difficulty | Notes |
| --- | --- | --- | --- | --- | --- |
| Single Region, Single AZ | ❌ None | ❌ Poor | ❌ Poor | ⭐ Easy | Not recommended for production |
| Single Region, Multi-AZ | ✅ Yes | ❌ Still regional | ❌ Still regional | ⭐⭐ Medium | Most common production setup |
| Multi-Region Active/Passive | Partial (DR) | ✅ Global reads | ❌ Writes centralized | ⭐⭐⭐ High | Best for DR + read scaling |
| Multi-Region Active/Active | ✅ Full | ✅ Best | ✅ Best | ⭐⭐⭐⭐ Very High | Hardest but best performance |

---

## Exam Tips

- **Active/Active** → Global reads AND writes
→ *DynamoDB Global Tables* is the typical example
- **Active/Passive** → Global reads only, write to single region
→ Matches many RDS/Multi-Region read replica scenarios
- **Multi-AZ ≠ Multi-Region**
Multi-AZ = HA
Multi-Region = global latency + DR
- If they mention **global low-latency writes**, the answer is **Active/Active**.

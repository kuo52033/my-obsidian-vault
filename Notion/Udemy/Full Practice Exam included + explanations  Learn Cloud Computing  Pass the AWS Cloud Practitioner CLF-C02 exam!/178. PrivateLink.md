AWS PrivateLink enables **private, secure connectivity** between VPCs and services **without exposing traffic to the public internet**.

It is used when you need to access a **service in another VPC** (your own or a vendor’s), but:

- You **don’t want VPC peering**
- You **don’t want to traverse the public internet**
- You need **scalable, per-service private access**

---

## **1. Why PrivateLink?**

### **VPC Peering problems**

- Not scalable (must connect VPC-to-VPC)
- Can’t selectively expose only one service
- Routing complexity increases
- All CIDR ranges must be non-overlapping

### **PrivateLink solution**

- Expose **a single service** privately
- No need for:
    - VPC peering
    - Internet Gateway
    - NAT Gateway
    - Route table changes

Traffic stays **within AWS private network**.

---

## **2. How PrivateLink Works**

### **Service Provider Side (e.g., SaaS vendor)**

- Runs the service in their own VPC
- Exposes it using a **Network Load Balancer (NLB)**
- Creates a **VPC Endpoint Service**

### **Service Consumer Side (you)**

- Creates an **Interface Endpoint (ENI)** inside your VPC
- Traffic from your applications goes to the ENI
- ENI connects privately to the provider’s NLB

### **Result**

- You access the vendor’s service **as if it were inside your VPC**
- No public IPs involved
- Fully private traffic

---

## **3. PrivateLink Use Cases**

- Accessing **SaaS vendor services** securely
(Datadog, Snowflake, Stripe, MongoDB Atlas, etc.)
- Sharing services across:
    - Different AWS accounts
    - Different VPCs
    - Different regions
- Offering an internal platform service to multiple business units

### Key idea:

> PrivateLink exposes services, not whole VPCs.

---

# ⭐ **4. PrivateLink vs VPC Peering**

| Feature | PrivateLink | VPC Peering |
| --- | --- | --- |
| Connectivity | **Service-level** | VPC-level (all subnets) |
| Uses public internet? | ❌ No | ❌ No |
| Requires IGW/NAT? | ❌ No | ❌ No |
| Requires route tables? | ❌ No | ✔ Yes |
| Scalable to many customers? | ✔ Very | ❌ Poor |
| Overlapping CIDR allowed? | ✔ Yes | ❌ No |
| Implementation | NLB + Interface Endpoint | Direct VPC-to-VPC |

**CLF Secret:** CIDR overlap is fine with PrivateLink, but not with VPC Peering.

---

# ⭐ **5. CLF-C02 Exam Tips**

### **1. PrivateLink definition**

> “Private access to a service in another VPC using an Interface Endpoint and NLB.”

### **2. When exam says:**

- "Need private, scalable access to SaaS or vendor service" → **PrivateLink**
- “Need to expose only one service, not whole VPC” → **PrivateLink**
- “CIDR overlap exists” → **PrivateLink**, NOT VPC peering
- “Do not want peering or internet” → **PrivateLink**

### **3. Not transitive**

PrivateLink is **point-to-point** per service.

### **4. Terminology to memorize**

- Provider = **Endpoint Service**
- Consumer = **Interface Endpoint**
- Load Balancer = **Network Load Balancer**

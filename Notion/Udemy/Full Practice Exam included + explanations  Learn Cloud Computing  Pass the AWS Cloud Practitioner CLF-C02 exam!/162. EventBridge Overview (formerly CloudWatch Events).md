## **1. What is EventBridge?**

**Amazon EventBridge** (previously called *CloudWatch Events*) is a **serverless event bus** that allows AWS services, SaaS partners, or your own applications to communicate using events.

- It helps you **react to events** occurring within AWS or external systems.
- Common use case: **build event-driven architectures**.

---

## **2. Core Use Cases**

### **A. Serverless Cron Jobs**

You can create scheduled rules using:

- **fixed rate** (e.g., every 15 minutes)
- **cron expressions**

Example:

- Rule: “Run every 1 hour”
- Target: Lambda function
→ Results in a **serverless cron job** (no servers to manage)

### **B. React to AWS Service Events**

EventBridge can detect when AWS resources change state.

Example:

- If someone logs in using the **root user**
- Forward the event to **SNS**
- Send an email notification to the security team

Other examples:

- EC2 instance state change
- S3 object creation event
- CodeBuild build status
- Trusted Advisor alert

---

## **3. Event Sources**

Event sources include:

- AWS services (EC2, S3, IAM, CodeBuild, etc.)
- **Scheduled events** (cron)
- **AWS Partner SaaS applications**
    - e.g., Zendesk, Datadog, Segment, Auth0
- **Custom applications**
    - You can publish your own custom events

---

## **4. Event Targets**

EventBridge can route events to many AWS services:

- **Lambda** (most common)
- **SNS**
- **SQS**
- **Step Functions**
- **Kinesis Streams / Firehose**
- **EC2 automation actions**
- **Glue / Batch / other compute services**

The routing is **fully managed**, meaning you don’t create queues or handle retries manually—EventBridge handles it.

---

## **5. Event Buses**

EventBridge provides multiple event buses:

### **1. Default Event Bus**

- Captures most AWS service events
- Captures scheduled events

### **2. Partner Event Bus**

- For receiving events directly from AWS SaaS partners

Example: Datadog → Partner Event Bus → your Lambda

### **3. Custom Event Bus**

- You create it yourself
- Used to send application-specific events
(microservices → event-driven apps)

---

## **6. Additional EventBridge Features**

### **Schema Registry**

- Automatically captures event structure (schema)
- Helps you understand event fields & data types
- Useful for developers when building integrations

### **Event Archive & Replay**

- You can archive events from an event bus
- You can **replay** events later
Useful for:
    - Debugging
    - Reprocessing failed workflows
    - Data recovery scenarios

# ⭐ **CLF-C02 Exam Tips (Important!)**

### **EventBridge vs SNS vs SQS**

EventBridge:

- Event routing service
- Filter events based on rules
- Connects AWS services, SaaS apps, and custom apps
- Supports cron scheduling
- Event-driven architecture

SNS:

- Pub/sub messaging
- Fan-out notifications
- No event filtering by payload (only topics)

SQS:

- Message queue
- Consumer pulls messages
- Supports retries & dead-letter queues

> If the exam says “react to AWS resource state changes,” choose EventBridge.

---

### **Default, Partner, and Custom Event Buses**

- **Default** → AWS services + schedules
- **Partner** → SaaS apps
- **Custom** → your own applications

> Expect exam questions comparing these three.

---

### **Root user login alert**

> Q: “How to send an email whenever root logs in?”
> ✔ IAM root login event → EventBridge → SNS → email

Super common CLF-C02 question.

---

### **Schedule Cron Jobs**

> If the task runs every hour / daily / weekly, the answer is:
> ✔ **EventBridge scheduled rule**

NO need for EC2 cron or Lambda timers.

---

### **Event Replay & Archiving**

- Know that EventBridge supports **event archiving** and **event replay**
- But deep configuration details are *not needed* for CLF-C02

## What is DocumentDB?

Amazon DocumentDB is a **fully managed NoSQL document database**, designed to be **compatible with MongoDB**.

Think of it as **Aurora but for MongoDB workloads**.

## Key Characteristics

- Document-oriented (stores semi-structured JSON-like data)
- MongoDB API–compatible
→ Existing MongoDB applications can run with minimal changes
- Storage automatically grows in **10 GB increments**
- Replication across **3 Availability Zones**
- Designed for **millions of requests per second**
- Fully managed: automated patching, backups, failover

---

## When to Use DocumentDB?

Use DocumentDB when:

- Your application uses **MongoDB**
- You store **JSON documents**
- You need flexible schemas (not relational)
- You want a fully managed MongoDB-compatible environment

---

## DynamoDB vs DocumentDB (Quick Comparison)

| Feature | DynamoDB | DocumentDB |
| --- | --- | --- |
| Database type | Key-value & document NoSQL | Document NoSQL |
| Schema | Flexible | Flexible (JSON) |
| API | DynamoDB API | MongoDB API |
| Serverless | Yes | No (provisioned instances) |
| Scaling | Automatic (fully serverless) | Scales storage automatically; instances scale manually |
| Access patterns | Ultra-low latency (ms) | Low latency, but higher than DynamoDB |
| Best for | High-scale serverless workloads | MongoDB migrations or document apps |

# 106. Neptune Overview

## What is Neptune?

Amazon Neptune is a **fully managed graph database** optimized for storing and querying **highly connected datasets**.

Graph databases store:

- **Nodes** (entities)
- **Edges** (relationships)
- **Properties** (metadata)

## Typical Graph Examples

- Social networks (users ↔ friends)
- Recommendation engines
- Fraud detection graphs
- Knowledge graphs (e.g., Wikipedia)

---

## Key Characteristics

- Purpose-built for **graph queries**
- Supports **Gremlin** and **SPARQL** graph query languages
- Replication across **3 AZs**
- Up to **15 read replicas**
- Millisecond latency queries
- Optimized for billions of relationships

---

## When to Use Neptune?

Choose Neptune when the question mentions:

- Graph relationships
→ friends, likes, connections, dependencies
- Complex traversal queries
→ “shortest path”, “mutual friends”, “recommend items”
- Knowledge graphs or fraud graph detection

---

## DynamoDB vs Neptune vs RDS (Quick Comparison)

| Feature | DynamoDB | Neptune | RDS |
| --- | --- | --- | --- |
| Data model | Key-value / document | Graph | Relational |
| Best for | Scale, low latency | Connected data | Structured tables |
| Query | Simple queries | Graph traversals | SQL |
| Example | User profiles | Social network | E-commerce DB |

# 107. Timestream Overview

## What is Timestream?

Amazon Timestream is a **serverless, fully managed time-series database** optimized for storing data that changes over time.

### What is time-series data?

Data indexed by time, such as:

- IoT sensor readings
- Application metrics
- Performance monitoring
- Financial time series
- Device telemetry

---

## Key Characteristics

- Purpose-built for **time-ordered data**
- Automatically scales storage & compute
- 1,000× faster and 1/10 the cost of relational databases for time-series workloads
- Built-in time-series analysis functions
- Serverless: no provisioning, no patching

---

## When to Use Timestream?

Choose Timestream when:

- The data is **time-based**
→ timestamps, metrics, continuous measurement
- You need high ingestion rates (millions of events/sec)
- Real-time analytics on IoT or operational data

---

## Timestream vs DynamoDB vs RDS (Quick Comparison)

| Feature | Timestream | DynamoDB | RDS |
| --- | --- | --- | --- |
| Purpose | Time-series | Key-value/document | Relational |
| Query style | Time-series functions | Simple lookups | SQL |
| Ideal for | Metrics, IoT, time trends | General workloads | Transactional apps |
| Scaling | Fully serverless | Fully serverless | Vertical/horizontal |

# 108. Managed Blockchain Overview

## What is Managed Blockchain?

Amazon Managed Blockchain is a fully managed service to create and operate **blockchain networks** without managing infrastructure.

## Key Characteristics

- Supports **Hyperledger Fabric** and **Ethereum**
- No need to manually configure blockchain nodes
- Decentralized architecture: multiple members share the ledger
- Secure and scalable blockchain network provisioning
- Reduces overhead for managing peer nodes and ordering services

---

## When to Use Managed Blockchain?

Choose this service when:

- You need a **decentralized ledger**
→ multi-party transactions without a central authority
- You need to integrate with **Ethereum** or **Hyperledger Fabric**
- You want tamper-resistant, immutable transaction records

---

## Blockchain vs Database (quick perspective)

| Feature | Blockchain | Database |
| --- | --- | --- |
| Architecture | Decentralized | Centralized |
| Data changes | Immutable, append-only | Mutable |
| Trust model | Multiple parties, no central control | Single owner/operator |
| Use cases | Supply chain, finance, multi-party workflows | Apps, analytics, storage |

## What is AWS Batch?

AWS Batch is a **fully managed batch processing service** that allows you to efficiently run **hundreds of thousands of batch jobs** on AWS, at any scale.

A **batch job** is:

- A job with a **clear start and end**
- Different from continuous / streaming jobs (which never stop)

Batch jobs are often used for:

- Nightly processing
- Scientific computing
- Large-scale data transformations
- Media rendering
- Image/video batch operations

---

## How AWS Batch Works

### ✔ Automatic Compute Provisioning

Batch **dynamically launches EC2 or Spot Instances** based on job volume.

### ✔ Job Submission

You submit jobs into a **Batch Job Queue**.

Batch chooses:

- When it runs
- How many instances to launch
- What instance type to use

### ✔ Runs Docker Containers

Each batch job is defined as:

- A **Docker image**
- A **task definition** (same concept used in ECS)

If it runs on ECS → it can run on Batch.

---

## Architecture Diagram (Simplified)

```plain text
S3 → Batch Job Queue → Batch-managed ECS/EC2 → Output to S3
```

Example use case:

User uploads images → Batch processes them → Outputs new results into S3.

Batch auto-scales the EC2 compute required.

---

## Why Use AWS Batch?

- Handles large workloads with **automatic scaling**
- Works with **Spot Instances** for major cost savings
- You focus on logic, not servers
- Supports **any runtime** that can run inside Docker
- No need to build your own job scheduler or autoscaler

---

## AWS Batch vs AWS Lambda

| Feature | AWS Lambda | AWS Batch |
| --- | --- | --- |
| **Execution Time Limit** | **15 minutes max** | **No limit** |
| **Runtime** | Only supported languages (Node, Python, Java, etc.) | **Anything** inside a Docker container |
| **Compute Model** | Fully serverless | Managed EC2 / Spot (NOT serverless) |
| **Disk Space** | Small `/tmp` (512MB–10GB) | Full EC2 storage (EBS / Instance store) |
| **Best For** | Lightweight, event-driven tasks | Long-running, heavy workloads |

### When to choose which:

- **Lambda** → short tasks, event-driven, serverless
- **Batch** → large jobs, long-running logic, custom runtimes, heavy processing

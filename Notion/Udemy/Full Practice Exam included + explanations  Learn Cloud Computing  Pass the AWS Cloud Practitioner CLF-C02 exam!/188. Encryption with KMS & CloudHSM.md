AWS provides multiple options to encrypt data **at rest** and **in transit**, using cryptographic keys stored in managed or customer-controlled systems.

---

# **1. Encryption Types**

## **A. Encryption at Rest**

Data stored on physical media:

- EBS volumes
- S3 objects
- RDS / Aurora databases
- Redshift
- EFS
- Glacier / Deep Archive

## **B. Encryption in Transit**

Data moving between systems:

- Client → AWS
- EC2 → DynamoDB
- EC2 → S3
- Between on-prem → AWS

### Goal:

> Ideally encrypt BOTH at rest & in transit.

---

# **2. Key Management – KMS**

KMS = **Key Management Service**, AWS’s centralized encryption and key management system.

### Key properties:

- Keys stored and managed by AWS
- Customer controls **who can use keys** via IAM policies
- Used across AWS services (EBS, S3, RDS, EFS, CloudTrail, etc.)

### Common services automatically encrypted:

- CloudTrail logs
- Glacier
- Storage Gateway

---

# **3. CloudHSM — Customer-Controlled Hardware Security**

CloudHSM provides a **dedicated hardware security module** (HSM) for you to manage encryption keys **completely by yourself**.

### Differences from KMS:

| Feature | KMS | CloudHSM |
| --- | --- | --- |
| Who manages keys? | AWS manages | **Customer manages** |
| Hardware | Shared, AWS controlled | **Dedicated hardware** HSM |
| Use cases | General encryption | Highest security/compliance |
| FIPS Level | 140-2 Level 2 | **140-2 Level 3** |
| Key visibility | You never see private keys | You fully control keys |

### When to choose CloudHSM:

- Regulatory/compliance requires **sole custody** of encryption keys
- Must meet **FIPS 140-2 Level 3**
- Need custom cryptographic operations

---

# **4. KMS Key Types**

## **A. Customer-Managed Keys (CMK)**

- You create & manage the key
- You define IAM access policies
- You control rotation (automatic yearly or manual)
- You can **import your own keys (BYOK)**

✔ Most flexible option

✔ Most common exam answer when customer wants control

---

## **B. AWS-Managed Keys**

- Keys created & managed automatically by AWS
- Named like `aws/s3`, `aws/ebs`, etc.
- Used when enabling simple “default encryption” in services

✔ Easiest option

✔ Good for basic encryption needs

---

## **C. AWS-Owned Keys**

- Used internally by AWS services
- You do **not** see or manage these keys
- Used for large-scale operations across accounts

✔ Very low friction

✔ Zero customer control

---

## **D. CloudHSM / Custom Key Store Keys**

- Keys stored inside your **CloudHSM cluster**
- Cryptographic operations performed in hardware
- You maintain complete custody of the keys

✔ Highest security level

✔ Required for strict compliance

---

# ⭐ **CLF-C02 Exam Tips**

### **1. KMS is the default answer for encryption**

If the exam says:

- “Encrypt S3 objects”
- “Encrypt EBS volumes”
- “Encrypt RDS instance”
- “Control access to encryption keys”

→ **Answer is KMS**

---

### **2. CloudHSM is NOT KMS**

Use CloudHSM when:

- Customer needs full control of encryption keys
- Compliance requires **FIPS 140-2 Level 3**
- Must *own and manage* encryption hardware

---

### **3. Customer-managed key vs AWS-managed key**

- Need control, key rotation, BYOK? → **Customer-managed key**
- Want simplest option? → **AWS-managed key**
- Key name starts with `aws/` → **AWS-managed**

---

### **4. Encryption at Rest vs In Transit**

- At rest: stored (EBS, S3, RDS)
- In transit: data moving across networks
- BOTH should be enabled for max security

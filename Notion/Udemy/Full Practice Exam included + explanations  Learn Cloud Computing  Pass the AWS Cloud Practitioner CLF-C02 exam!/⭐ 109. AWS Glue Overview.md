AWS Glue is a **fully managed ETL (Extract, Transform, Load) service** used to prepare data for analytics, machine learning, and reporting.

**Exam keyword:** “ETL” → Think **AWS Glue**

**Also keyword:** “serverless data preparation”

---

## 🧩 How ETL Works (Concept Diagram)

```plain text
[ Data Sources ] → (Extract) → [ Glue ] → (Transform) → [ Target System ]
     S3, RDS                           |                     Redshift, S3
                                       | (Load)

```

---

## Key Features

- **Serverless**: No infrastructure to manage
- **ETL Jobs**: Python/Scala scripts automatically generated or manually written
- **Connectors** for S3, RDS, DynamoDB, Redshift, JDBC sources
- **Integrated** with Athena, Redshift, EMR

---

## Example Workflow (Simple)

1. Extract structured/unstructured data from S3 or RDS
2. Clean, normalize, join, or enrich the datasets
3. Load the transformed data into Redshift for analytics

---

## When to Use AWS Glue

Use Glue when:

- You need large-scale **data preparation** for analytics
- Data must be cleaned/formatted before loading into Redshift
- You want **serverless ETL** without managing servers
- You work with a data lake (S3) and need to build transformation pipelines

---

## Glue ETL vs Glue Data Catalog

| Feature | Glue ETL | Glue Data Catalog |
| --- | --- | --- |
| Purpose | Transform data | Metadata store for datasets |
| Serverless | Yes | Yes |
| Usage | Build ETL pipelines | Used by Athena, EMR, Redshift Spectrum |
| Contains | Scripts, jobs | Table schemas (columns, types, partitions) |

# ⭐ 110. AWS DMS Overview (Database Migration Service)

## What is AWS DMS?

AWS DMS is a fully managed **database migration service** used to migrate data:

- From **on-premises → AWS**
- From **AWS → AWS**
- Between **database engines** (heterogeneous migration)

**Exam keyword:** “Migrate database” → Answer is **AWS DMS**

---

## Migration Architecture Diagram

```plain text
[ Source Database ]
         ↓
   (Extract via DMS)
         ↓
   [ DMS Replication Instance ]
         ↓
   (Transform/Replicate)
         ↓
   [ Target Database ]
```

---

## Key Features

- **Source database remains online** during migration
(Minimal downtime)
- Supports **one-time migrations** and **continuous replication**
- Built-in resiliency and self-healing replication
- Supports multiple engines:
    - Oracle, MySQL, PostgreSQL
    - SQL Server, MariaDB
    - DynamoDB, S3, Redshift
    - Aurora, etc.

---

## Types of Migrations

### **1. Homogeneous Migration**

Same engine → same engine

Examples:

- MySQL → MySQL
- Oracle → Oracle

✔ Minimal transformation required

---

### **2. Heterogeneous Migration**

Different engine → different engine

Examples:

- Microsoft SQL Server → Aurora
- Oracle → PostgreSQL

✔ DMS + AWS Schema Conversion Tool (SCT)

✔ Used when schema types are incompatible

---

## When to Use DMS

Use AWS DMS when:

- You must migrate data **with minimal downtime**
- You are moving from on-prem → AWS
- You are changing database engines (heterogeneous)
- You need ongoing **replication** (not just one-time copy)

---

## DMS vs Glue (Quick Comparison)

| Feature | AWS DMS | AWS Glue |
| --- | --- | --- |
| Purpose | Database migration | ETL (data transformation) |
| Keeps source online? | Yes | Not applicable |
| Transformations | Minimal | Extensive (cleaning, joining, enriching) |
| Best for | Moving DBs | Preparing data for analytics |

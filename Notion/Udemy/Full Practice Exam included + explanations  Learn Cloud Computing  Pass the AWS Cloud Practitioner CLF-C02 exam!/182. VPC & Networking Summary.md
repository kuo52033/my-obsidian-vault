This summarizes all major VPC & networking concepts for AWS CLF-C02.

---

# **1. VPC Basics**

- **VPC (Virtual Private Cloud)** = your isolated network in AWS
- **Subnets**:
    - Tied to **one Availability Zone**
    - Public or Private depending on route table

---

# **2. Internet Access**

### **Internet Gateway (IGW)**

- Provides **public** internet access
- Attached at **VPC level**
- Used by **public subnets**
- Route table: `0.0.0.0/0 → IGW`

### **NAT Gateway / NAT Instance**

- Allows **private subnets** to access the internet
- Does **not** allow inbound internet traffic
- NAT Gateway is managed; NAT Instance is self-managed

---

# **3. Security Layers**

### **Network ACL (NACL)**

- **Subnet-level**, stateless
- ALLOW + DENY rules
- Must allow return traffic explicitly

### **Security Group (SG)**

- **Instance-level**, stateful
- Only ALLOW rules
- Automatically allows return traffic
- Can reference **other SGs**

---

# **4. IP Addressing**

- **Public IPv4** changes on stop/start
- **Private IPv4** persists
- **Elastic IP** = fixed public IPv4 (charged)
- **IPv6** = always public, free

---

# **5. VPC Connectivity**

### **VPC Peering**

- Connect two VPCs **privately**
- Must have **non-overlapping** CIDR blocks
- **Not transitive**
(A ↔ B, B ↔ C → A cannot reach C)

### **VPC Endpoints**

- Private access to AWS services without internet
- **Gateway Endpoint** → S3, DynamoDB
- **Interface Endpoint (PrivateLink ENI)** → almost all other services

### **PrivateLink**

- Private, service-level access across VPCs
- Provider exposes service via **NLB**
- Consumer connects through **Interface Endpoint**

### **VPC Flow Logs**

- Capture network traffic metadata from:
    - VPC
    - Subnets
    - ENIs
- Helps troubleshoot connectivity issues

---

# **6. Hybrid Connectivity**

### **Site-to-Site VPN**

- Encrypted connection over **public internet**
- Fast to set up
- Requires:
    - **Customer Gateway (CGW)** (on-prem)
    - **Virtual Private Gateway (VGW)** (AWS)

### **Direct Connect (DX)**

- Dedicated private physical connection
- High bandwidth, stable latency
- Slow setup (weeks/months)
- More costly

### **Client VPN**

- Users/laptops connect securely to VPC
- Based on OpenVPN
- Access private EC2 / VPC resources remotely

---

# **7. Transit Gateway (TGW)**

- Central hub to connect:
    - Multiple VPCs
    - VPN
    - Direct Connect
    - On-prem
- Scales to thousands of networks
- Supports transitive routing
- Ideal for large, multi-account architectures

# ⭐ **CLF-C02 Ultimate Decision Guide**

| Need | Correct Service |
| --- | --- |
| Public internet access | Internet Gateway |
| Private subnet needs outbound internet | NAT Gateway |
| Block an IP at subnet level | NACL |
| Allow inbound only from another instance’s SG | Security Group |
| Connect 2 VPCs | VPC Peering |
| Private access to AWS services | VPC Endpoint |
| Private access to S3/DynamoDB | **Gateway Endpoint** |
| Private access to any other AWS service | **Interface Endpoint (PrivateLink)** |
| Private access to vendor SaaS | PrivateLink |
| Capture network traffic info | VPC Flow Logs |
| Connect on-prem to AWS quickly | Site-to-Site VPN |
| High-bandwidth dedicated link | Direct Connect |
| User laptop → AWS VPC | Client VPN |
| Connect many VPCs / hybrid networks | Transit Gateway |

AWS security is based on the **Shared Responsibility Model**, which defines what AWS secures vs. what *you* must secure.

---

# **1. AWS Responsibility — “Security OF the Cloud”**

AWS is responsible for protecting the **infrastructure that runs AWS services**.

### **AWS Handles:**

- Physical hardware (servers, storage, networking)
- Global infrastructure:
    - Regions
    - Availability Zones
    - Edge locations
- Managed services infrastructure:
    - S3
    - DynamoDB
    - RDS (underlying EC2 + OS + patching)
    - Lambda infrastructure
- Virtualization layer & hypervisor
- Physical access controls
- Service availability & durability guarantees

> You do not patch the OS for RDS, DynamoDB, S3, Lambda, etc.
> AWS does.

---

# **2. Customer Responsibility — “Security IN the Cloud”**

You must secure **how you use AWS services**.

### **You Handle:**

- **Your data**
- **Application code**
- **Identity & Access Management (IAM)**
- **Network configuration**
    - Security Groups
    - NACLs
- **Operating system management** (for EC2)
    - Install updates
    - Patch OS packages
    - Configure firewall
- **Encryption choices**
    - Enable S3 bucket encryption
    - Enable RDS encryption at creation
    - Manage IAM permissions for access
- **Client-side encryption**
- **Server-side encryption controls**
- **Resource-level settings**
    - Public/private access settings
    - Bucket policies
    - RDS parameter groups

---

# **3. Shared Controls (Both Parties Contribute)**

Some tasks are shared between AWS and the customer.

### **Examples of Shared Controls**

- **Patch management**
    - AWS patches managed services
    - You patch EC2 OS
- **Configuration management**
    - AWS manages infrastructure config
    - You configure services (S3 buckets, security groups, etc.)
- **Awareness & training**
    - AWS trains their staff
    - You train your employees on cloud usage

---

# **4. Service-Specific Examples**

## **A. RDS Responsibility Split**

### **AWS handles:**

- Underlying EC2 instances
- DB engine patching
- OS patching
- No SSH access allowed
- Automated backups & maintenance

### **Customer handles:**

- DB security groups (network access)
- DB user accounts / permissions
- DB parameter groups
- Enabling DB encryption
- Deciding whether DB is public or private

---

## **B. S3 Responsibility Split**

### **AWS handles:**

- Infinite storage capacity
- Separation between accounts
- AWS employees cannot access your content
- Encryption availability (when enabled)

### **Customer handles:**

- Bucket policies & IAM permissions
- Access control (public/private)
- Enabling encryption (SSE-S3, SSE-KMS, etc.)
- Configuring lifecycle & versioning
- Protecting data from unintended public access

---

# **5. Visual Summary**

### **Customer Responsibilities**

- Data
- Application code
- IAM
- OS on EC2
- Network configs (SG, NACL)
- Encryption configuration
- Client-side data protection

### **AWS Responsibilities**

- Compute/storage/network hardware
- Regions / AZs / Physical infra
- Managed service infrastructure
- Underlying OS for managed DBs
- Virtualization layer

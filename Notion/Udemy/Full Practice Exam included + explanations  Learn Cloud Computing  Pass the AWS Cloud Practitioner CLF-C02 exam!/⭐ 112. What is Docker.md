### **High-level Overview**

Docker is a **software development platform** used to **package applications into containers**.

A *container* bundles your application **plus all its dependencies**, ensuring it runs **the same way everywhere**, without compatibility issues.

---

## **Why Docker? – Key Benefits**

- **Environment consistency**
→ Same behavior on any machine, OS, or cloud provider.
- **Fast scaling / fast startup**
→ Containers launch in **seconds**, ideal for modern cloud environments.
- **Language & OS agnostic**
→ Works with Java, Node.js, Python, MySQL, Redis… anything.
- **Simplified deployment**
→ Developers ship a container; Ops runs it anywhere.

---

## **Docker on EC2 (Example Architecture)**

On a single EC2 instance, you can run multiple isolated containers:

- A Java app container
- A Node.js API container
- A MySQL database container
- A Redis cache container

Each container is isolated but shares the same host resources.

---

## **Docker Images**

To run a container, you need a **Docker image**, which defines:

- Application code
- Runtime (Node, Java, Python…)
- Dependencies
- OS libraries

These images are stored in **Docker registries**.

### **Image Registries**

- **Docker Hub (public)** — OS images, language runtimes, databases
- **Amazon ECR (private)** — secure, private image storage for AWS workloads

---

## **Docker vs. Virtual Machines**

### **Traditional VM (EC2)**

```plain text
Infrastructure → Host OS → Hypervisor → Guest OS → App


```

- Each VM includes a **full OS**
- Heavyweight
- Slower to start
- Less efficient

### **Docker Architecture**

```plain text
Infrastructure → Host OS → Docker Daemon → Containers


```

- Containers **share the host OS**
- Much more lightweight
- Start in seconds
- Better resource utilization

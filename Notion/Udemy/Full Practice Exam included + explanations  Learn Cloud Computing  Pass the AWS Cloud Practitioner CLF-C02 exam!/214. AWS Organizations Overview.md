## **1. What is AWS Organizations?**

- A **global service** to centrally manage **multiple AWS accounts**.
- Structure:
    - **Management Account** (formerly "Master Account")
    - **Member Accounts** (Child Accounts)
- Common exam themes: **Consolidated Billing**, **SCP (Service Control Policies)**, **multi-account strategy**.

---

## **2. Key Benefits**

### ✅ **Consolidated Billing**

- All accounts share **one bill**, billed to the management account.
- No need to add payment methods to each member account.

### ✅ **Volume Discounts Across Accounts**

- Aggregated usage = more discounts for:
    - EC2
    - S3
    - Data transfer
- Reserved Instances (RI) & Savings Plans **shared** across accounts.

### ✅ **Automated Account Creation**

- Organizations API allows programmatic creation of new accounts
(e.g., sandbox accounts).

### ✅ **Central Governance with SCP**

- Restrict what services **CAN or CANNOT** be used in specific accounts/OUs.
- Applies to **all users & roles**, including **root user** of the member account.
- **Does NOT apply** to:
    - Service-linked roles
    - Management account

---

## **3. Multi-Account Strategy**

Reasons to split workloads into separate AWS accounts:

- ✔ Per **department**
- ✔ Per **environment** (dev / staging / prod)
- ✔ Per **project**
- ✔ For **regulatory isolation**
- ✔ To apply **different SCP restrictions**
- ✔ Separate **service quotas**
- ✔ Centralized **logging account**

Best practices:

- Enable **CloudTrail** in all accounts, send logs to a **central logging account**.
- Standardize tagging for billing/chargeback.
- Use OUs to group accounts logically.

## **4. Organizational Units (OUs)**

Example structure:

```plain text
Root
 ├─ Dev OU
 │    ├─ Dev-App
 │    └─ Dev-Analytics
 ├─ Prod OU
 │    ├─ HR
 │    └─ Finance
 └─ Sandbox OU


```

You can nest OUs to represent:

- Environments
- Teams
- Compliance boundaries

## **5. Service Control Policies (SCPs)**

SCPs = Organization-wide guardrails.

### **Key Points**

- Apply at **OU or account level**.
- Affect **all IAM users & roles**, including **root**.
- Must include **explicit Allow** statements.
(Deny overrides Allow.)
- Do NOT apply to the **management account**.

### **Common Use Cases**

- Restricting non-compliant services (PCI, HIPAA, etc.)
- Preventing use of expensive services (e.g., EMR in dev accounts)
- Enforcing least-privilege at the organization level

---

## **6. SCP Examples**

### ❌ **Blacklist DynamoDB**

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Deny",
      "Action": "dynamodb:*",
      "Resource": "*"
    }
  ]
}


```

### ✅ **Whitelist Only EC2 + CloudWatch**

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "ec2:*",
        "cloudwatch:*"
      ],
      "Resource": "*"
    }
  ]
}


```

## **7. Exam Tips (High Frequency)**

- Organizations is **global**, CloudTrail is **per region**.
- SCP **does not affect the management account**.
- SCP can **block even root user** of a member account.
- Consolidated billing enables **volume discounts** + **RI sharing**.
- Use Organizations API for **automated account provisioning**.
- Use multi-account strategy for **isolation**, **compliance**, **cost control**.

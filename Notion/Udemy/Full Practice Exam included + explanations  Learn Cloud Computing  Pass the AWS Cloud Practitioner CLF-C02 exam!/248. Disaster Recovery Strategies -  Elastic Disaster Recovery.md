## **What is Disaster Recovery (DR)?**

- Strategies to **recover applications and data after a disaster**
- Trade-off between **cost**, **recovery time**, and **complexity**

---

## **DR Strategies (From Cheapest → Most Expensive)**

### **1. Backup and Restore (Cheapest)**

- Data is **backed up** to the cloud
- Application **is NOT running**
- Restore data and rebuild infrastructure after disaster

**Characteristics**

- ✅ Lowest cost
- ❌ Longest recovery time (RTO)
- ❌ Manual restore required

**Exam Tip**

> Cheapest DR strategy = Backup and Restore

---

### **2. Pilot Light**

- Only **core components** (e.g. database) are running
- Minimal setup in the cloud
- Application servers are NOT running

**Characteristics**

- ⚖️ Low cost
- ⚖️ Faster than backup & restore
- Requires scaling DB and launching app servers during recovery

---

### **3. Warm Standby**

- **Full application stack** is running
- At **minimum capacity**
- Scale up when disaster happens

**Characteristics**

- ⚖️ Medium cost
- ⚡ Faster recovery
- Infrastructure already running

---

### **4. Multi-Site / Hot Site (Most Expensive)**

- Full application **running at full capacity**
- Traffic can be switched immediately

**Characteristics**

- ❌ Highest cost
- ⚡⚡ Fastest recovery
- Near zero downtime

**Exam Tip**

> Most expensive DR strategy = Multi-Site / Hot Site

---

## **Cost vs Availability Summary**

| Strategy | Cost | Recovery Speed |
| --- | --- | --- |
| Backup & Restore | Lowest | Slowest |
| Pilot Light | Low | Slow–Medium |
| Warm Standby | Medium | Fast |
| Multi-Site | Highest | Fastest |

---

## **Multi-Region Disaster Recovery**

- Applications deployed in **multiple regions**
- **Route 53** used to fail over traffic
- Example:
    - Primary: `us-east-1`
    - Failover: `eu-west-2`

**Exam Keywords**

- *Region failure*
- *Route 53 failover*
- *Multi-region recovery*

---

## **Exam Memory Hook**

> Cheapest → Backup & Restore
> **Fastest → Multi-Site**

# **249. AWS Elastic Disaster Recovery (DRS)**

## **What is AWS Elastic Disaster Recovery (DRS)?**

- Fully managed **disaster recovery service**
- Formerly **CloudEndure Disaster Recovery**
- Enables fast recovery of:
    - Physical servers
    - Virtual machines
    - Cloud-based servers

---

## **How AWS DRS Works**

1. Install **replication agent** on source servers
2. Perform **continuous block-level replication**
3. Data is replicated into a **low-cost staging area** in AWS
4. On disaster:
    - Launch **production EC2 instances**
    - Use proper instance sizes and EBS volumes
5. Optional **failback** to original environment

---

## **What Gets Replicated**

- Operating systems
- Applications
- Databases
- Disk data (block-level)

---

## **Key Features**

- Continuous replication
- Very low RPO (near-zero data loss)
- Fast recovery (minutes)
- Supports ransomware recovery
- On-premises → AWS
- Cloud → AWS

---

## **Common Use Cases**

- Data center failure
- Critical databases (Oracle, MySQL, SQL Server)
- Enterprise apps (SAP)
- Ransomware protection

---

## **AWS DRS vs Traditional DR**

| Feature | Traditional DR | AWS DRS |
| --- | --- | --- |
| Replication | Manual / Complex | Automatic |
| Cost | High | Low (staging) |
| Recovery Speed | Slow | Fast (minutes) |

---

## **Exam Keywords → Correct Answer**

If the question mentions:

- Continuous block-level replication
- On-premises to AWS recovery
- Fast failover within minutes
- CloudEndure

👉 **Answer: AWS Elastic Disaster Recovery (DRS)**

> AWS DRS = continuous replication + fast disaster recovery
